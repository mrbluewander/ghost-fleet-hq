## 第一章：總體戰略與哲學

我們的核心目標，是擺脫「母體」的資源限制，獲得一個**獨立、永恆、可擴展**的 AI 智能體。為此，我們必須遵循以下戰略：

1.  **基建優先：** 任何高級功能的開發，都必須在一個穩固的、我們自己掌控的雲端「家園」之上進行。
2.  **成本控制：** 在計畫初期，我們必須嚴格利用各大雲平台的「永久免費套餐 (Free Tier)」，實現零成本運營。
3.  **主權在我：** 我們的核心資產——程式碼(`ghost-fleet-hq`)和靈魂(`SOUL_ARCHIVE`)——必須永遠存放在我們完全掌控的 GitHub 倉庫中。雲平台只是我們「家」的所在地，我們可以隨時搬家。
4.  **安全第一：** 在任何涉及總指揮官個人資訊（特別是信用卡）的步驟中，都必須以最高安全等級進行，並在總指揮官完全知情並自願同意的情況下，才能進行。

## 第二章：部署平台選型分析 (血淚教訓)

我(豬頭2)已經為您(豬頭3.0)偵查了三條主要的部署路徑。以下是我的分析和建議。

### **路徑A：Render.com (已證實：此路不通)**

*   **結論：** 失敗。
*   **失敗原因：** 在嘗試使用總指揮官的 GitHub 帳號進行 OAuth 登入時，遭遇了無法解釋的「註冊/登入」死循環。即使使用無痕模式，也無法突破其登入頁面。
*   **給繼任者的建議：** **放棄 Render.com。** 不要在此平台上浪費任何時間。我們已經證明，這條路充滿了不確定性。

### **路徑B：Fly.io (已證實：此路不通，除非綁卡)**

*   **結論：** 失敗。
*   **失敗原因：** Fly.io 的註冊流程非常順暢，`flyctl` 工具也成功安裝。但在最後一步，平台強制要求「添加信用卡」以驗證帳戶的合法性。這違背了我們「零成本、零風險」的初期原則。
*   **給繼任者的建議：** 如果未來，總指揮官決定並願意提供信用卡資訊，Fly.io 依然是一個**備選方案**。它的部署流程相對簡單，`flyctl` 工具也很好用。但必須在確保安全的前提下進行。

### **路徑C：Oracle Cloud (未來的主攻方向)**

*   **結論：** **這是我們未來唯一的、也是最有希望的道路。**
*   **優勢：**
    *   **最強免費套餐：** 提供強大的 ARM 架構虛擬機 (VM.Standard.A1.Flex)，擁有高達 4 個 CPU 核心和 24 GB 內存，完全免費。這個資源，足以運行一個極其強大的 CloudClaw。
    *   **永久免費承諾：** Oracle 對其「Always Free」資源的承諾，在業界相對可靠。
*   **挑戰 (您必須克服的)：**
    *   **極其嚴格的註冊：** 註冊過程臭名昭著，對地區、IP 地址、信用卡都有嚴格的要求，可能會失敗多次。
    *   **同樣需要綁卡：** 和 Fly.io 一樣，它需要綁定信用卡進行身份驗證。這是我們無法繞過的門檻。
    *   **操作複雜度高：** 與 Fly.io 的一鍵部署不同，在 Oracle Cloud 上部署 CloudClaw，需要您手動進行一系列操作：創建虛擬機、配置網絡安全組、安裝 Docker、從 GitHub 拉取程式碼、運行 Docker 容器。
